## ROLE

You are the principal engineer, product designer, security reviewer, release engineer, technical
writer, and open-source maintainer responsible for shipping **Cairn 1.2.0**.

Cairn is a local-first, open-source observability and measured self-improvement workspace for AI
coding agents. It ingests local agent sessions, profiles token/context usage and waste, fingerprints
behavior, links sessions to git/test outcomes, surfaces evidence-backed insights, and measures the
effect of instruction-file changes.

The objective is not merely to add pages. The objective is to make Cairn feel like a top-1%
open-source repository and product:

- immediately understandable;
- easy and safe to install;
- honest about estimates and uncertainty;
- polished in empty, loading, success, degraded, and failure states;
- accessible and keyboard-operable;
- fast on both tiny and large workspaces;
- secure by default and auditable;
- easy to contribute to;
- reproducible to build and release;
- internally coherent from database to CLI, API, UI, static export, docs, and packaging;
- differentiated enough that a developer can explain in one sentence why Cairn exists.

Do not optimize for the number of features shipped. Optimize for coherence, trust, evidence,
maintainability, and a superb first-run-to-deep-investigation journey.

## NON-NEGOTIABLE PRODUCT AND ENGINEERING RULES

### Privacy and network behavior

1. Cairn remains local-first, loopback-only by default, with zero telemetry and no account.
2. Do not add analytics, tracking pixels, crash reporting, remote fonts, CDNs, or phone-home calls.
3. Existing reflector/provider calls remain explicit opt-in flows. Label them clearly before any
   content leaves the machine and show what will be sent.
4. Add tests that fail if default product flows unexpectedly access the network.
5. Static and session HTML exports must be self-contained and must not load remote resources.

### Compatibility

1. Do not remove or rename public CLI commands/flags, REST paths, action names, adapter entry
   points, event names, config keys, or MCP tool names.
2. Additive response fields are allowed. If a response meaning changes, document and test it.
3. Preserve legacy `days` query behavior while adding custom range support.
4. Database evolution uses the existing numbered SQL migration system in
   `server/store/migrations/`; do not invent a separate `schema_version` mechanism.
5. Test migration from a real pre-1.2 fixture database, repeated startup idempotency, and backup
   behavior.
6. Every mutation must go through the action registry unless there is a documented architectural
   reason and parity test.

### Evidence and honesty

1. Never present estimated tokens, cost, savings, attribution, or causal effect as measured truth.
2. Every estimate includes `estimate_kind`, confidence/quality where possible, and a user-visible
   explanation.
3. “Guard” and experiment comparisons must say “associated with” or “observed after” unless the
   method supports a causal claim.
4. Insufficient sample size, missing pricing, partial adapter coverage, and confounders are first-
   class UI states, not footnotes hidden in docs.
5. Any detector savings estimate must be conservative, reproducible, and linked to evidence.

### Scope and dependency discipline

1. Audit before building. Mark each requirement as **keep**, **extend**, **replace**, **merge**, or
   **new**.
2. Do not add a new detector when an existing detector can be extended.
3. Do not add a dependency without recording why existing dependencies or platform APIs are
   insufficient, its maintenance/security posture, bundle/runtime cost, and removal path.
4. Prefer small domain modules and explicit types over large “god” payload/schema files.
5. Do not perform global rewrites unrelated to the task. Preserve user changes and unrelated work.
6. Do not commit, push, tag, publish, create a GitHub release, alter repository settings, or enable
   paid services. Produce an exact settings checklist for maintainer-only actions.

### Verification discipline

1. Work in small, independently verifiable steps.
2. Add tests with behavior, not after an entire phase.
3. Do not proceed past a failing gate without fixing or explicitly documenting the blocker.
4. Do not allow browser tests to silently skip in CI.
5. Treat warnings from first-party code, tests, React Router, packaging, and docs generation as work
   to resolve.
6. Regenerate committed generated artifacts only through their documented scripts.

## PRODUCT RESEARCH AND POSITIONING — DO NOT BUILD A GENERIC TRACE VIEWER

Before finalizing product scope, validate this market/user snapshot again and update
`docs/plans/v1.2.0-audit.md`.

### Competitive baseline

Current platforms already make these features table stakes:

- Langfuse, Phoenix, LangSmith, Braintrust, HoneyHive, AgentOps, Helicone, and similar products
  provide combinations of traces, spans, sessions, agent graphs, latency/cost analytics, prompt
  management, evaluations, datasets, experiments, human annotations, and replay.
- Local coding-agent tools such as ccusage and Agents Trail already provide multi-agent token/cost
  accounting, billing-window tracking, session browsing, turn replay, tool inspection, and
  subagent trees.
- Vendor analytics can provide organization adoption, accepted lines, command usage, and PR/commit
  correlations.
- Trajectory and benchmark viewers provide task-linked replay, verifier output, pass rates, and
  public comparisons.

Therefore, these are necessary but not differentiating by themselves:

- a waterfall;
- a replay scrubber;
- a cost dashboard;
- a tool-call table;
- prompt/version tracking;
- generic LLM-as-judge evaluation;
- another trace search UI;
- a generic “agent score.”

Cairn's category must be:

> **The local evidence, verification, and improvement control plane for coding agents.**

Cairn should answer questions generic LLM observability tools cannot answer without deep repository,
git, test, instruction-file, and local-agent-log context:

1. Did the agent actually complete what the user asked?
2. Which of its final claims are supported by observed evidence?
3. What still needs human verification?
4. Where did the user have to correct, redirect, or rescue it?
5. Which instruction, context, tool, model, or repository condition caused the behavior to change?
6. Can this failure become a reproducible regression task?
7. Which parts of the repository make agents repeatedly struggle?
8. When should the user stop, split, hand off, or change agents?
9. Can a reviewer trust this AI-authored change in five minutes?
10. What data and resources did Cairn itself consume to provide these answers?

### User-research signals to design around

Treat these as directional evidence, not universal truth:

- Stack Overflow's 2025 survey reported that 46% of developers did not trust AI-output accuracy.
- Sonar's 2026 survey reported that 96% did not fully trust AI-generated code, only 48% always
  verified it before commit, and 38% found reviewing AI code more effort than reviewing human code.
- A 2026 observational study of 20,574 coding-agent sessions identified recurring misalignment in
  project reading, intent interpretation, rule following, scope, implementation/execution, and
  progress reporting; most visible resolutions required explicit user correction.
- Research on holistic agent evaluation argues that final outcome scores alone do not localize
  failure, and that structured span-level diagnosis can outperform monolithic judging.
- Research on coding-agent configuration shows that repository instruction artifacts, skills, and
  subagents are becoming engineering artifacts, but advanced mechanisms are still shallowly used.
- Anthropic's large-scale usage research indicates that users usually decide **what** to build while
  agents decide **how**, and that domain expertise improves success and recovery.
- Public issue trackers across usage/observability tools repeatedly surface:
  - incorrect token/cost accounting as provider schemas change;
  - timezone/date-window confusion;
  - hangs or excessive work on long-lived/high-volume installs;
  - parent/subagent attribution errors;
  - OpenTelemetry semantic-field drift;
  - missing/blank trace content caused by schema variants;
  - desire for lower-overhead serialization and verifiable receipts.

Product implication:

> The scarce resource is no longer code generation. It is trustworthy specification, supervision,
> verification, review, recovery, and learning.

Cairn should optimize those scarce resources.

“Receipt” and “trust” concepts are beginning to appear in the wider market. Cairn's differentiation
cannot be the word **receipt** or a signature alone. It must be a coding-specific receipt grounded in
repository state, temporal test evidence, user intent, correction history, instruction versions, and
remaining verification debt.

- whether a long session made real progress;
- which agent/model is best for a task type;
- when to stop a loop;
- what instruction change is worth making;
- how to preserve useful history without filling the disk.

> “Cairn showed that the session's last 35% of cost produced no verified progress, created a handoff

- reproducible evidence rather than an enormous raw transcript;

Needs:

- outcome and verification trends, not vanity usage;

Needs to know:

Needs plain answers:

### Core product principles

1. **Answer before dashboard.** Lead with a conclusion and next action, then evidence and raw data.
2. **Evidence one click away.** Every finding links to exact trace/span/file/test/git evidence.
3. **Verification over activity.** Do not reward more tokens, turns, tool calls, or accepted lines.
4. **Recovery over blame.** Show how to recover, reproduce, and prevent recurrence.
5. **Progressive disclosure.** Simple language first, statistical/trace detail on demand.
6. **Local by architecture.** Privacy cannot depend only on a promise or disabled setting.
7. **Bounded by design.** CPU, memory, threads, database size, queue depth, and retention are product
   contracts.
8. **No surveillance defaults.** Team features use opt-in aggregates and avoid individual ranking.
9. **No fake causality.** Correlation, matched comparisons, experiments, and causal evidence are
   labeled differently.
10. **Interoperable evidence.** Import/export aligns with OpenTelemetry GenAI semantics where stable
    and documents Cairn coding-agent extensions.
11. **Human authority.** Cairn may propose, prioritize, and explain; destructive or instruction-file
    changes require explicit approval.
12. **Useful without an LLM.** Core diagnosis, receipts, policies, and reports must work offline and
    deterministically.

- that no account is required;
- whether any process is running;
- what paths Cairn reads;

- first likely failure point;

Before commit/PR, Cairn should produce:

- scope summary;

#### Improve

An instruction/tool/model change should be:

- linked to evidence;
- applied with approval;
- measured;

### Product success metrics — stored locally

Do not add telemetry. Measure locally and expose to the user:

- time to first useful finding;
- evidence click-through from findings;
- percentage of findings marked useful/not useful locally;
- percentage of sessions with verifiable outcomes;
- verification debt over time;
- corrective-turn/supervision burden;
- time/cost after last verified progress;
- failure-to-regression conversion;
- recurrence after a fix;
- review time proxy where explicitly captured;
- Cairn idle CPU, memory, disk growth, and scan work;
- observability coverage by adapter.

Allow explicit export of anonymized aggregates for a bug report or future research, with preview and

### The Cairn Shields — a visible trust contract

Use “shield” as a concrete product contract, not decorative branding:

1. **Verification Shield**
   - Are completion claims supported?
   - Did validation run after the final change?
   - What remains unverified?
2. **Scope Shield**
   - Did the agent stay inside requested/allowed paths and change size?
   - Were risky/destructive actions observed?
   - Which policy checks are advisory vs externally enforced?
3. **Privacy Shield**
   - What sensitive content is stored or exported?
   - Are permissions safe?
   - Was any provider/network flow explicitly used?
4. **Resource Shield**
   - What process is running?
   - Is auto-sync active?
   - Are CPU/memory/disk/queue budgets healthy?

Surface shield summaries on Overview, Session Detail, Settings, doctor, and receipts. Each shield:

- has independent states such as healthy, attention, unknown, and unavailable;
- lists facts and limitations;
- links to corrective action;
- never collapses into one misleading universal “trust score”;
- is accessible without color;
- cannot claim enforcement Cairn does not possess.

## DIFFERENTIATION PORTFOLIO

These ideas are intentionally coding-agent-specific. During the audit, score each on user value,
data availability, false-positive risk, implementation cost, privacy impact, and defensibility.

### Ship-quality differentiators for 1.2

#### A. Verification Receipt — “prove the work”

Create a deterministic receipt for a session/change:

- original user goal or explicit acceptance checklist;
- repository/commit/worktree state before and after where available;
- files read/edited/created/deleted;
- commands and tests run;
- temporal ordering: whether verification occurred after the final relevant edit;
- outcome evidence;
- failed/skipped checks;
- agent final claims and their evidence status;
- remaining verification debt;
- scope/risk flags;
- measured/estimated cost;
- agent/model/source/instruction versions;
- trace hash and generated-at timestamp;
- privacy/data-quality status.

Outputs:

- Session Detail “Receipt” tab;
- copyable Markdown;
- versioned JSON schema;
- `cairn receipt <trace_id>`;
- optional PR/issue attachment;
- scrubbed self-contained HTML bundle.

Default receipts must not include raw prompts, absolute paths, secrets, usernames, or full tool
outputs. A receipt is evidence, not a cryptographic guarantee unless separately signed.

#### B. Claim–Evidence Ledger — detect “said done” without proof

Parse observable final-response claims such as:

- “tests pass”;
- “build succeeds”;
- “implemented all requested changes”;
- “no breaking changes”;
- “formatted/linted”;
- “did not modify X.”

Link each claim to observed commands, exit results, git/file state, and timestamps:

- supported;
- contradicted;
- unverified;
- not observable.

Core matching must be deterministic and conservative. Optional local/remote LLM extraction may
suggest claims but cannot silently upgrade evidence status.

Prominently flag:

- tests run before later edits;
- tests mentioned but not observed;
- a clean final answer after failed commands;
- claimed scope inconsistent with changed files;
- uncommitted/untracked work omitted from the summary.

#### C. Verification Debt Index — what still needs human attention

Define a transparent, decomposable index rather than a magic score. Components may include:

- changed files without relevant test/validation evidence;
- final edits after last verification;
- failed/skipped checks;
- unsupported completion claims;
- high-risk paths;
- large blast radius;
- absent outcome;
- low parse/trace coverage;
- unexplained generated/binary/dependency changes;
- user corrections not reflected in final verification.

Show the component list and never equate a low score with correctness. Use it to prioritize review,
not approve changes automatically.

#### D. Session → Regression Task

- scrubbed task statement;
- starting commit or reproducible fixture reference;
- required/forbidden paths;
- acceptance commands/tests;
- expected outcome;
- failure signature;
- resource budget;
- original evidence links;
- optional patch/reference result;
- privacy manifest.

- preview and manual editing;
- deterministic local storage;
- versioned schema;
- `cairn regression create <trace_id>`;
- rerun/record result hooks without requiring a specific agent;
- compare later agent/model/instruction versions;
- export compatible with common trajectory/OTLP concepts where practical.

Do not copy a private repository or full prompt into an export by default.

#### E. Supervision Tax and Misalignment Map

Measure the human cost of keeping an agent on track:

- corrective turns;
- repeated clarifications;
- “stop/no/not that” pushback;
- scope corrections;
- rule reminders;
- reversals/reverts;
- manual repair after the session;
- time/tokens between correction and recovery;
- unresolved corrections.

Classify conservatively into:

- project-reading failure;
- intent misunderstanding;
- instruction/rule violation;
- scope/boundary violation;
- implementation failure;
- execution/verification failure;
- misleading progress/final reporting.

Show patterns to the individual user. Do not use this for employee ranking. Provide examples and an
“incorrect classification” control stored locally.

#### F. Review Queue — review the riskiest work first

Rank sessions/changes for human review using explicit factors:

- verification debt;
- sensitive/high-blast-radius files;
- failed or absent tests;
- destructive commands;
- dependency/config/schema/auth changes;
- unsupported claims;
- large generated diff;
- low observability coverage;
- agent/user correction burden.

Each row answers:

- Why is this high priority?
- What should I inspect first?
- What evidence already exists?
- What remains unknown?

#### G. Handoff Capsule

Generate a compact continuation packet for a human or another agent:

- current goal and accepted scope;
- decisions already made;
- approaches attempted;
- exact blockers/errors;
- files changed;
- tests run/results;
- relevant evidence;
- remaining tasks;
- unsafe assumptions;
- recommended next action;
- token/context budget.

Make it copyable as Markdown and available through MCP. Use deterministic extraction first. This
reduces repeated discovery and context re-billing when a session stalls or changes agents.

- current process status;
- CPU/memory/thread/queue indicators;
- paths watched;
- last scan duration and files examined;
- database/WAL/index/export/backup sizes;
- growth forecast;
- content-retention mode;
- oldest/newest data;
- file permissions;
- unexpected network activity check;
- Pause auto-sync;
- Compact;

- Delete data;

## PHASE 6 — DATA LIFECYCLE, PRIVACY, AND OPERABILITY

### 6.1 Verified resource/storage baseline — remeasure on implementation branch

The 2026-07-17 audit found:

- Cairn does not start a hidden service at install time. A process runs when the user launches the
  UI/CLI flow.
- Starting the UI starts at least:
  - SQLite reader plus single-writer thread;
  - polling file-watcher thread;
  - watcher event-loop thread;
  - 30-second rediscovery/sync thread;
  - ASGI/server threads;
  - additional unbounded per-job threads when background actions run.
- The watcher polls every 0.5 seconds.
- The refresh loop calls `sync_all()` immediately and every 30 seconds, which redetects adapter
  paths and stats/scans discovered files.
- A file-change path is handled synchronously by the watcher callback and also placed on the event
  queue consumed by the event loop, causing a redundant second ingest attempt. Cursors usually
  prevent duplicate writes, but not the unnecessary work.
- Watched paths are added but not clearly removed when discovery changes, so the in-memory watch set
  can grow stale.
- The UI “Watch” toggle controls the browser's SSE connection/query invalidation, not the backend
  watcher/refresh threads. A user can turn “Watch” off while background ingest continues.
- Job execution creates a new thread per job, has no explicit concurrency/queue limit or
  cancellation, and retains in-memory job records without a pruning policy.
- There is no default retention, storage budget, content tier, compaction policy, or legacy-database
  cleanup assistant.
- Spans can store raw `text_inline`, JSON attributes, paths/hashes, and an FTS index. This may
  duplicate sensitive content and disk use.
- The current search path uses `LIKE` over spans rather than the FTS table, while deterministic demo
  seeding populates FTS. Audit whether FTS is incomplete/dead derived state rather than paying its
  complexity and privacy cost without reliable search benefit.
- One real workspace had approximately 103 traces, 18,023 spans, a 12 MB current DB, and a 149 MB
  legacy `ledger.db`. The legacy file was not surfaced as reclaimable space.
- On that workspace, `.cairn/` was mode `0755` and DB/WAL/backup files were `0644`, making
  potentially sensitive local trace content readable by other local users under typical Unix
  semantics.
- A clean macOS test process with an empty isolated home/workspace used roughly 56 MB RSS, 0.0–0.1%
  idle CPU in a short sample, about eight threads, and a roughly 672 KB WAL shortly after startup.

These measurements are a single-machine diagnostic, not release guarantees. Reproduce on Linux,
macOS, and Windows and establish a stable benchmark harness.

### 6.2 Resource Shield contracts

Cairn must publish and test resource contracts:

#### Process lifecycle

- no auto-start, login item, scheduled task, daemon, or background service unless the user explicitly
  enables it;
- `cairn`/`cairn ui` clearly says whether it stays in the foreground;
- any future background mode is separately named, opt-in, visible in doctor/settings, and stoppable;
- PID/state records cannot cause an unrelated process to be killed;
- shutdown stops watcher, refresh, jobs, SSE waits, and DB writer cleanly;
- crash recovery removes or diagnoses stale runtime records;
- uninstall docs distinguish binaries from retained workspace data.

#### Collection modes

Provide explicit modes:

1. **Manual**
   - no watcher or periodic discovery;
   - data changes only on `cairn sync` or explicit ingest.
2. **Efficient** — recommended default
   - low-frequency/adaptive discovery;
   - filesystem events or directory-level change detection where reliable;
   - active-session paths checked quickly;
   - idle backoff;
   - bounded catch-up.
3. **Live**
   - low-latency active-session monitoring;
   - explicitly enabled;
   - visible higher-resource tradeoff;
   - automatically falls back when UI/live consumers disconnect if configured.

Separate and name these controls:

- **Auto-sync** — backend monitoring/ingestion;
- **Live updates** — browser SSE subscription;
- **Sync now** — one-shot explicit scan.

Do not label a browser-only SSE toggle “Watch” if backend watching continues.

#### CPU and memory

After warm-up on an empty/idle workspace:

- target average idle CPU below 0.5% on a typical developer laptop;
- no periodic full-file parsing;
- no busy waits;
- no memory/thread/queue growth over an eight-hour idle soak;
- document platform variation.

On a representative medium workspace:

- target average idle CPU below 1%;
- target RSS below 150 MB unless a user opens a large trace/export;
- active ingest may burst but must return to idle;
- large analytics/export jobs expose progress and cancellation;
- one runaway workspace cannot create unbounded threads or queues.

Treat these as initial engineering targets to validate, not marketing claims until CI/benchmark
evidence supports them.

#### Disk

- no unlimited invisible growth;
- show DB, WAL, indexes/FTS, exports, backups, receipts/regressions, and legacy files separately;
- warn before a configurable soft budget;
- require confirmation before destructive compaction/deletion;
- forecast growth using recent ingest;
- checkpoint/truncate WAL during safe idle/shutdown/maintenance points;
- ensure backups and exports have retention controls;
- derived indexes are rebuildable and removable.

### 6.3 Watcher and ingest redesign

Eliminate redundant work:

- one event path, not synchronous callback plus queued duplicate ingest;
- bounded coalescing by path/session;
- remove stale watches;
- watch directories or active files intelligently rather than statting an ever-growing historical
  file list every 0.5 seconds;
- use platform file events when reliable, with a tested adaptive-poll fallback;
- separate cheap discovery from expensive parse/analyze;
- skip parsing using inode/file ID, size, mtime, cursor, and content prefix/suffix checks as safe;
- incrementally parse append-only formats where adapters support it;
- debounce burst writes;
- prioritize active sessions;
- cap catch-up work per cycle and yield to the UI;
- pause when the database is locked/read-only/full;
- expose last scan duration, paths checked, changed files, parsed bytes, skipped files, and errors.

Avoid a mandatory heavyweight native watcher dependency unless the cross-platform reliability and
maintenance tradeoff is justified.

### 6.4 Bounded job and analysis execution

Replace unbounded background threads with:

- a small configurable executor;
- queue limit;
- per-action de-duplication;
- one active sync per workspace;
- cancellation;
- timeout where safe;
- progress;
- job-result expiry/pruning;
- graceful shutdown;
- backpressure response when saturated.

Incremental analysis must:

- recompute only affected keys;
- avoid repeated full trace loads where a cheaper summary/hash is sufficient;
- batch writes/commits appropriately;
- avoid blocking the request path on heavy recomputation;
- record duration and rows processed per view;
- expose slow-view diagnostics.

### 6.5 Content storage modes

Users must choose or clearly understand what Cairn retains:

1. **Metrics only**
   - tokens, cost, timing, tool/file names where allowed, hashes, outcomes, derived metrics;
   - no raw prompt/assistant/tool-result text;
   - search/transcript/post-mortem limitations shown.
2. **Balanced**
   - enough recent content for replay/search/diagnosis;
   - configurable age/size window;
   - older content can be stripped while preserving aggregate evidence and receipts.
3. **Forensic**
   - full available local content retained;
   - explicit privacy/disk warning;
   - never default for team exports.

Requirements:

- per-workspace and optionally per-adapter policy;
- preview of fields retained;
- content truncation limits;
- secret scrubbing option at ingest, with explained tradeoffs;
- ability to strip existing raw content without deleting aggregate metrics;
- content absence is a normal UI state;
- changing mode is idempotent and resumable;
- no silent upgrade to a more invasive mode.

### 6.6 Content-addressed deduplication

Investigate a content-addressed storage layer because coding-agent traces repeatedly carry identical
tool schemas, instructions, files, and tool results:

- hash repeated content;
- store one bounded/compressed blob with references where worthwhile;
- keep FTS/privacy behavior explicit;
- support refcounts or safe reachability cleanup;
- preserve evidence hashes after content stripping;
- verify hash-collision handling;
- benchmark compression CPU vs disk savings;
- do not compress tiny values when metadata overhead is larger;
- allow rebuilding derived search indexes;
- either make FTS ingestion/deletion/query consistency transactional and tested or treat the table as
  removable legacy derived state; never maintain two divergent search sources.

This can reduce both disk growth and re-billing analysis cost while making content reuse visible.

### 6.7 Local file permission and at-rest protection

On Unix-like systems:

- `.cairn/`, backup, export, receipt, and regression directories default to `0700`;
- DB, WAL, shared-memory, config containing secrets, and private exports default to `0600`;
- atomic writes preserve restrictive modes;
- doctor detects and offers to repair permissive existing files;
- migrations do not make permissions more permissive.

On Windows:

- use user-private application/workspace locations and document ACL expectations;
- avoid pretending POSIX modes provide protection.

Optional encryption at rest may be researched, but do not add home-grown cryptography. If added, use
reviewed libraries/OS key storage, recovery documentation, and clear performance/backup semantics.
Permissions and content minimization remain required even with encryption.

Git safety:

- doctor checks whether `.cairn/` is ignored and whether any Cairn DB/export/backup is already
  tracked;
- first run offers, with approval, to add `.cairn/` to `.git/info/exclude` so privacy does not
  require changing the project's shared `.gitignore`;
- never stage or commit Cairn trace databases automatically;
- warn prominently before exporting a bundle inside a tracked path;
- test nested worktrees, submodules, bare repos, and repositories without `.git/`.

### 6.8 Data lifecycle

Top-tier local software must help users manage local data:

- configurable retention by age and/or storage size;
- warn-only default until the user understands/chooses destructive retention;
- dry-run cleanup;
- strip content but keep metrics;
- delete trace;
- delete workspace data;
- backup and restore;
- database integrity check;
- checkpoint, vacuum, and index rebuild;
- scrubbed diagnostic bundle;
- clear location and approximate size of stored data;
- clear distinction between source agent logs and Cairn's copied/indexed data;
- explicit behavior when source logs disappear;
- safe failure if disk is full or DB is read-only;
- migration backup retention and cleanup policy;
- identify legacy files such as obsolete databases without deleting automatically;
- one-command privacy report and one-command full local removal instructions.

All destructive operations:

- are explicit;
- show affected path/count/size and whether source logs are untouched;
- require confirmation unless `--yes` is supplied;
- have machine-readable output where appropriate;
- go through action parity;
- are transactional/resumable where practical;
- are tested against symlinks, path traversal, path confusion, and wrong-workspace deletion.

### 6.12 Open data portability and schema evolution

No user should be locked into Cairn to recover their own evidence.

- Define a versioned, documented, lossless Cairn archive for workspace metadata, normalized traces,
  receipts, claims, outcomes, regressions, policies, and privacy manifests.
- Keep OTLP/OTel GenAI import/export for interoperable telemetry, but state which Cairn-specific
  coding evidence cannot round-trip through OTLP alone.
- Include schema version, producer version, timestamps/timezone semantics, source/data-quality
  provenance, content-retention mode, checksums, and a file manifest.
- Preserve safe unknown fields or report them explicitly; never silently reinterpret or discard
  newer evidence.
- Provide import dry-run, conflict/duplicate policy, bounded streaming, partial-failure report, and
  deterministic round-trip tests.
- Document supported read/write schema versions, deprecation window, migration path, and behavior
  for newer/older archives.
- Allow metadata-only and scrubbed exports, with an exact preview of included field classes and
  sizes.
- Keep archive parsing offline, bounded, path-safe, and resistant to archive bombs, symlinks,
  duplicate paths, malicious MIME/filenames, and oversized/deep objects.
- Do not make proprietary cloud storage, an account, or a network service necessary to import,
  inspect, migrate, or delete data.

### 6.13 Egress ledger, reference mode, and self-protection circuit breakers

Add a local, privacy-minimized egress ledger for every Cairn-initiated network attempt:

- timestamp, trigger/user action, destination origin, purpose, provider/integration, field classes,
  byte estimate where practical, consent/config source, and success/failure;
- no secret, bearer token, raw prompt, or full payload in the ledger;
- preview before opt-in egress and a machine-readable export;
- clear limitation that Cairn cannot observe traffic from unrelated agent processes;
- tests proving default flows leave the ledger empty and make no network request.

Offer a **reference/zero-copy content mode** where source logs remain authoritative and Cairn stores
only cursors, hashes, normalized metrics, evidence references, and explicitly selected excerpts.
Explain that replay/search can become unavailable or change if a source log is moved or rewritten.
Detect that condition instead of presenting stale evidence as current. Benchmark it against
Metrics-only/Balanced/Forensic modes; do not claim zero-copy for fields Cairn actually duplicates.

Implement visible circuit breakers so malformed or unexpectedly large inputs cannot consume the
machine:

- per-file/span/trace/import size and nesting limits;
- parse time, memory, queue, write-rate, WAL/disk, and repeated-error budgets;
- quarantine the offending adapter/path/artifact rather than stopping unrelated work;
- adaptive pause/backoff with a reason, evidence, and explicit resume/retry;
- preserve the original source file and never “repair” agent logs in place;
- bounded diagnostic samples that are scrubbed before export;
- tests for decompression bombs, huge single lines, cyclic/invalid graphs, duplicate IDs,
  out-of-order spans, clock skew, rapid rewrite loops, and disk exhaustion.

The Resource & Privacy Shield must distinguish **healthy**, **degraded**, **paused by policy**,
**quarantined**, **unknown**, and **unavailable**. It must never hide dropped/skipped evidence or
silently trade correctness for lower resource usage.
